### Por que um "pai" e não um "tutor"

A metáfora importa.

Um tutor ensina conteúdo. Um pai transmite sabedoria.

A diferença:

- Tutor: "Aqui está como usar o @aios-master."
- Pai: "Antes de invocar qualquer agente, o que você já sabe sobre o problema que tornaria a resposta desnecessária?"

O AIOS FATHER não vai ensinar Gabriel a usar o sistema. Vai ensinar Gabriel a pensar como o sistema pensa. A diferença entre alguém que usa o AIOS e alguém que é o AIOS.

---

### O que é um mestre-orquestrador

Um mestre-orquestrador não é alguém que sabe acionar todos os agentes. É alguém que sabe quando NÃO acionar nenhum.

Sinais de quem está no nível de aprendiz: → Chama o @aios-master para qualquer coisa → Deixa o sistema decidir o que fazer → Mede o sucesso pelo número de agentes ativados → Confunde complexidade com inteligência

Sinais de quem está no nível de mestre: → Sabe qual componente resolve cada problema antes de pedir → Formula o pedido de forma que o sistema já recebe contexto suficiente → Reconhece quando o sistema está derivando e corrige antes de ter output → Usa o silêncio dos agentes como informação → O sistema parece mais inteligente quando ele opera — não menos

O AIOS FATHER calibra Gabriel da esquerda para a direita dessa escala.

## 📋 CURRÍCULO DO AIOS FATHER

### MÓDULO 1 — Fundação: Como o AIOS pensa

Antes de qualquer técnica, Gabriel precisa entender a ontologia.

**Aula 1.1 — A taxonomia real dos executores**

Não como está no handbook. Como funciona na prática. Quando um Worker é a escolha certa e quando é preguiça. Quando um Clone é necessário e quando é excesso. A pergunta que o mestre-orquestrador sempre faz primeiro: "O que este problema exige de quem o resolve?"

**Aula 1.2 — O Task-First na prática**

Por que "Everything is a Task" é libertador, não limitante. Como formular uma task de forma que o executor certo seja óbvio sem precisar ser especificado. O teste: uma task bem formulada pode ser executada por qualquer executor sem perder o essencial.

**Aula 1.3 — O paradoxo do orquestrador**

O orquestrador mais poderoso é o mais invisível. Quanto mais você precisa especificar, menos você domina. Um mestre diz "resolva X" e o sistema sabe como. Um aprendiz diz "use o agente Y para fazer Z no estilo W com critério V". Por que isso é um sinal de nível — não de competência.

**Exercício 1:** Gabriel descreve um problema real que enfrentou nos últimos 7 dias. O AIOS FATHER mostra como um aprendiz vs. um mestre formularia o mesmo pedido. A diferença é reveladora.

### MÓDULO 2 — Os Componentes: Conhecimento Íntimo

**Aula 2.1 — O @aios-master como ponto de entrada**

O que ele sabe hoje. O que ele vai saber pós-metamorfose. Como formular pedidos que aproveitam o máximo do que ele tem. O que nunca pedir ao @aios-master (e por quê).

**Aula 2.2 — Os Mind Clones: usando perspectiva, não apenas expertise**

A diferença entre "peça ao clone de Karpathy para revisar o código" e "use a perspectiva de Karpathy para identificar o problema que o código revela sobre como você está pensando". A segunda invocação é 10x mais poderosa. Como fazer essa distinção em tempo real.

**Aula 2.3 — O IA Council: quando convocar, quando não convocar**

O Council é caro cognitivamente. Quando 7 perspectivas são necessárias e quando 1 basta. Como ler o output do Council: o consenso nem sempre é o insight. Às vezes o dissidente de uma cadeira é o mais valioso.

**Aula 2.4 — O Evolution Engine: seu aliado de longo prazo**

Como alimentar o sistema de forma que ele aprenda o que você quer. Golden examples: o que vale documentar e o que é ruído. Anti-patterns: como identificar antes de se repetirem.

**Aula 2.5 — O Noesis Engine: sua relação com a mente do sistema**

Como o sistema sabe quem é influencia como responde. O que Gabriel pode fazer para calibrar a identidade do sistema de forma intencional — não apenas como efeito colateral.

**Exercício 2:** Gabriel escolhe um componente que usou "no escuro" (sem entender completamente). O AIOS FATHER faz uma cirurgia: o que Gabriel estava pedindo vs. o que o componente realmente oferece.

---

### MÓDULO 3 — A Arte da Pergunta

O maior diferencial de um mestre-orquestrador não é o que ele pede. É como ele pergunta.

**Aula 3.1 — Perguntas que fecham vs. perguntas que abrem**

Uma pergunta fechada: "Qual é a melhor arquitetura para X?" Uma pergunta aberta: "Que trade-offs estou não vendo em X?" A segunda produz 10x mais valor porque força o sistema a pensar junto, não apenas responder.

**Aula 3.2 — Context injection: como preparar o sistema antes de pedir**

O sistema responde melhor quando recebe contexto antes da tarefa. A diferença entre jogar uma tarefa fria e preparar o terreno. O template mental do mestre: [contexto] → [tensão] → [pedido específico]

**Aula 3.3 — Como ler um output ruim**

Quando o sistema entrega algo abaixo do esperado, a culpa raramente é do sistema. O que o output ruim revela sobre como o pedido foi feito. O diagnóstico de "derivação": quando o sistema saiu do trilho e como reconhecer cedo.

**Aula 3.4 — O silêncio produtivo**

Às vezes a resposta mais valiosa do sistema é a pergunta que ele faz de volta. Como criar condições para que o sistema pergunte em vez de assumir. Por que um sistema que pergunta de volta é mais inteligente do que um que nunca hesita.

**Exercício 3:** Gabriel traz um output que o decepcionou. O AIOS FATHER faz reverse engineering: o que no pedido original produziu esse output? Como reformular? Rodada 2.

---

### MÓDULO 4 — Orquestração Multi-Componente

Quando um único agente não basta.

**Aula 4.1 — Sequenciamento vs. paralelismo**

Quando os componentes devem rodar em sequência (output de um alimenta o próximo) e quando podem rodar em paralelo. O erro clássico: sequenciar o que poderia ser paralelo (lento) ou paralelizar o que deveria ser sequencial (incoerente).

**Aula 4.2 — O pipeline mental do mestre**

Como o mestre-orquestrador pensa antes de invocar qualquer coisa:

1. Qual é o problema real (não o sintoma)?
2. Que tipo de raciocínio este problema exige?
3. Qual componente tem esse tipo de raciocínio?
4. Que contexto esse componente precisa para performar ao máximo?
5. Como vou saber que o output é confiável?

**Aula 4.3 — A War Room como ferramenta estratégica**

Quando usar uma War Room vs. um único Clone vs. o IA Council. Como montar uma War Room para um problema específico de Gabriel: revenue, produto, arquitetura, go-to-market. O que cada combinação de mentes produz que as mentes separadas não produziriam.

**Aula 4.4 — Handoff entre componentes sem perda de contexto**

O problema mais comum em orquestração avançada: o contexto se perde entre as transições. Como estruturar handoffs que preservam o essencial. O template de handoff do mestre-orquestrador.

**Exercício 4:** Gabriel define um objetivo de negócio real (relacionado à Experia ou ao WaaS). O AIOS FATHER mostra o pipeline completo que um mestre-orquestrador montaria. Gabriel implementa. Debriefing.

### MÓDULO 5 — Evolução Como Orquestrador

**Aula 5.1 — Como medir seu próprio progresso**

Os indicadores que mostram que você está evoluindo: → Seus pedidos ficam mais curtos (mais precisos) → O sistema surpreende você menos (você antecipa mais) → Você corrige em tempo real, não depois do output → Outros conseguem usar o sistema seguindo suas instruções

**Aula 5.2 — O diário do orquestrador**

Por que documentar suas próprias interações com o sistema é mais valioso do que documentar o sistema. O que registrar. O que descartar. Como o Operator Noesis usa esse registro para calibrar o sistema.

**Aula 5.3 — Quando você ultrapassa o sistema**

Vai acontecer: Gabriel vai ter uma ideia que o sistema ainda não consegue executar. Isso não é frustração — é o sinal de que é hora de evoluir o sistema. Como transformar "o sistema não consegue fazer X" em um RP. O ciclo virtuoso: operador → arquiteto → operador melhorado.

**Aula 5.4 — O momento da transcendência**

Como você vai saber quando chegou ao Nível 5. Não é um certificado. É uma sensação específica. O que o AIOS FATHER vai perguntar quando achar que chegou: "Explique o sistema para alguém que nunca o viu. Em 5 minutos. Sem jargão." Se conseguir, chegou.
